# DefectFlow 모바일 테스트 연동 가이드

## 목적
- 모바일웹 테스트 페이지에 `신고하기` 버튼을 붙여 DefectFlow의 `모바일 간편 등록` 화면으로 바로 연결하는 방법을 정리합니다.
- 운영 페이지, 테스트 베드, 사내 QA 페이지에서 동일한 방식으로 재사용할 수 있는 최소 연동 기준을 제공합니다.

## 적용 대상
- 모바일웹 또는 모바일 해상도 전용 테스트 페이지
- 캡처 없이 빠르게 결함을 등록하고 싶은 QA 흐름
- `FO (MO)` 화면 경로를 기준으로 결함을 정리하는 모바일 테스트 시나리오

## 관련 파일
- 위젯 스크립트: `docs/defectflow_mobile_report_widget.js`
- 등록 화면: `index.html?mode=mobile#register`
- 예제 페이지: `test_view/mobile_test_bed.html`
- 기존 공통 연동 가이드: `docs/테스트대상_연동가이드.md`

## 동작 방식
1. 모바일 페이지에 `defectflow_mobile_report_widget.js`를 삽입합니다.
2. 사용자가 우측 하단 `신고하기` 버튼을 누릅니다.
3. 위젯이 현재 페이지 URL, 제목, 기본 테스트 정보로 `pending_defect` 데이터를 구성합니다.
4. DefectFlow의 `?mode=mobile#register` 화면을 새 탭 또는 현재 탭으로 엽니다.
5. 등록 화면은 저장된 데이터를 읽어 모바일 간편 등록 폼을 채웁니다.

## 빠른 연동 예시

```html
<script>
window.DEFECTFLOW_MOBILE_REPORTER_CONFIG = {
  defaultScreenPath: 'FO (MO) > 메인화면 > 슬라이딩배너, 중분류',
  defaultTestType: '통합테스트',
  defaultSeverity: 'Minor',
  defaultPriority: 'Medium',
  envTag: 'ZeroStore Mobile QA'
};
</script>
<script src="https://mohenz.github.io/defect_manage/docs/defectflow_mobile_report_widget.js"></script>
```

## 권장 삽입 위치
- 테스트 대상 페이지의 `</body>` 바로 위에 삽입합니다.
- SPA라면 공통 레이아웃 또는 모바일 전용 진입 템플릿에 한 번만 넣습니다.
- 동일 페이지에서 스크립트가 중복 로드되더라도 위젯은 한 번만 생성됩니다.
- 위젯이 실행 버튼 아이콘용 `Font Awesome 6.4.0` 스타일시트를 자동 로드하므로 별도 아이콘 CSS를 추가하지 않아도 됩니다.

## 설정 값

### 런타임 설정
`window.DEFECTFLOW_MOBILE_REPORTER_CONFIG`로 아래 값을 제어할 수 있습니다.

- `buttonLabel`: 버튼 접근성 라벨. 기본값 `신고하기`
- `registerUrl`: 등록 화면 URL. 기본값 `https://mohenz.github.io/defect_manage/?mode=mobile#register`
- `defaultTestType`: 기본 테스트 구분. 기본값 `통합테스트`
- `defaultSeverity`: 기본 심각도. 기본값 `Minor`
- `defaultPriority`: 기본 우선순위. 기본값 `Medium`
- `defaultScreenPath`: 기본 화면 경로. 예: `FO (MO) > 메인화면 > 슬라이딩배너, 중분류`
- `defaultTitle`: 기본 결함 제목
- `creator`: 기본 등록자명
- `openInNewTab`: 새 탭 열기 여부. 기본값 `true`
- `envTag`: 환경 식별 문자열. 기본값 `Mobile Report Widget`

### meta 태그 설정
런타임 설정이 없으면 아래 meta 태그 값을 읽습니다.

```html
<meta name="defectflow:screen-path" content="FO (MO) > 메인화면 > 슬라이딩배너, 중분류">
<meta name="defectflow:test-type" content="통합테스트">
<meta name="defectflow:severity" content="Minor">
<meta name="defectflow:priority" content="Medium">
<meta name="defectflow:title" content="모바일 결제 버튼 미노출">
<meta name="defectflow:creator" content="홍길동">
```

적용 우선순위는 아래와 같습니다.
1. `window.DEFECTFLOW_MOBILE_REPORTER_CONFIG`
2. `meta[name="defectflow:*"]`
3. 스크립트 기본값

## 전달 데이터
위젯은 아래 정보를 `localStorage.pending_defect`로 저장합니다.

- `title`
- `steps_to_repro`
- `menu_name`
- `screen_name`
- `screen_url`
- `screenshot`
- `env_info`
- `test_type`
- `severity`
- `priority`
- `creator`

참고:
- 모바일 위젯은 `screenshot`을 비워 둡니다.
- `defaultScreenPath`는 `menu_name`과 `screen_name`으로 분리되어 저장됩니다.
- `env_info`에는 `envTag`와 브라우저 `User Agent`가 함께 기록됩니다.

## 현재 제약사항
- 현재 모바일 공통 위젯은 `localStorage` 기반 전달만 사용합니다.
- 따라서 테스트 페이지와 DefectFlow 등록 화면이 같은 origin일 때 가장 안정적으로 동작합니다.
- 다른 origin의 외부 사이트에서 `registerUrl`만 GitHub Pages로 열면 `pending_defect`가 자동 전달되지 않을 수 있습니다.
- cross-origin 모바일 연동이 필요하면 별도 브리지 구현 또는 기존 데스크톱 위젯/확장프로그램 방식 검토가 필요합니다.

## 모바일 등록 화면 기준
- 등록 URL은 반드시 `?mode=mobile#register` 형식을 사용합니다.
- `mode=mobile`이면 사이드바가 숨겨지고 모바일 간편 등록 폼이 열립니다.
- 결함 발생 화면 선택 목록은 `SCREEN_PATH` 공통코드 중 `FO (MO)` 항목만 표시됩니다.

## 검증 체크리스트
1. 페이지 우측 하단에 `신고하기` 버튼이 보이는지 확인합니다.
2. 버튼 클릭 시 모바일 등록 화면이 열리는지 확인합니다.
3. 결함 제목, 테스트 구분, 심각도, 우선순위, URL이 기본값으로 채워지는지 확인합니다.
4. `결함 발생 화면` 목록이 `FO (MO)` 항목만 보여주는지 확인합니다.
5. 저장 후 DefectFlow 목록에 신규 결함이 생성되는지 확인합니다.

## 문제 해결

### 버튼이 보이지 않음
- 스크립트가 `</body>` 앞에서 한 번만 로드되는지 확인합니다.
- 페이지에 `document.body`가 생성되기 전에 커스텀 초기화 코드를 막고 있지 않은지 확인합니다.

### 등록 화면은 열리지만 값이 비어 있음
- same-origin 환경인지 먼저 확인합니다.
- `window.DEFECTFLOW_MOBILE_REPORTER_CONFIG`가 스크립트보다 먼저 선언됐는지 확인합니다.
- `defaultScreenPath` 형식이 `A > B > C` 구조인지 확인합니다.

### 화면 경로 선택값이 없음
- DefectFlow의 `SCREEN_PATH` 공통코드에 `FO (MO)` 데이터가 등록되어 있는지 확인합니다.

### 저장이 안 됨
- 로그인 상태인지 확인합니다.
- 제목은 5자 이상 입력해야 합니다.
- 필수 항목인 `결함 제목`, `결함 발생 화면`, `결함 내용 또는 재현 절차`, `테스트 구분`, `심각도`, `우선순위`가 모두 입력됐는지 확인합니다.

## 운영 권장 사항
- 모바일 테스트 페이지마다 `defaultScreenPath`를 다르게 주어 초기 선택 부담을 줄입니다.
- `envTag`에 서비스명 또는 채널명을 넣어 나중에 결함 출처를 구분합니다.
- 운영 반영 전에는 예제 페이지 `mobile_test_bed.html`로 먼저 검증합니다.
